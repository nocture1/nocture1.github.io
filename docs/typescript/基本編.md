# 基本編
## 型の定義
変数には明示的に他を割り当てることができる。

```ts
const a: number = 1

const b: string = "abc"

const c: boolean = true
```

明示的に型を割り当てない場合、初期値と値の再割り当て可能性により型が推論される。

```ts
// 1型
const a = 1

// abc型
const b = "abc"

// true型
const c = true
```

特定の値を表現するのが**リテラル型**。  
一方で、値の再割り当てを許す `let` による宣言では状況が異なる。

```ts
// number型
let a = 1

// string型
let b = "abc"

// boolean型
let c = true
```

値の再割り当てが行える状況では、  
「初期値と互換性を持つ値であれば再割り当てが可能」である。  
このため、 `let` による宣言では初期値よりも広い型に割り当てられる。  
これを**型の拡大**という。

ここで、値を割り当てない場合はまた状況が異なる。

```ts
// any型
let a
```

`any` は型に対するヒントを持たないため、本質的に動的型付けとなる。  
TypeScriptのメリットを捨てることになるため、できるだけ `any` を宣言しないようにする。

複雑なオブジェクトでも同様に型の推論が働く。

```ts
// {id: number}型
let a = {id: 3}

// number[]
let b = [2, 3, 4]

// (number | boolean | string)[]
let c = [2, true, "typescript"]
```

## ユニオン型
前節の通り複数ある型のうちいずれかを表現するのが**ユニオン型**。  
まず、TypeScriptでは型を定義するには `type` で指定する。

以下のもっとも単純なユニオン型の定義はint Enumと同等になる。

```ts
type a = 1 | 2 | 3
```

TypeScriptでは `Enum` もあるが、型安全性などの問題もあり推奨されていない。  
ユニオン型ではリテラル型や基本型を組み合わせることもできるほか、  
配列やオブジェクトリテラルとも組み合わせることができる。

```ts
type b = 1 | "typescript"
type c = number | string
type d = 1 | string
type e = string[] | boolean | 42
type f = {id : 42} | "string"
```

ユニオン型は先頭でパイプを記述できるため、インデントを崩すことなく記述できる。

```ts
type g = 
    | { id: number }
    | { name: string }
    | { admin: boolean }
```

## 関数・引数の型

引数は変数と同様に型が割り当てられる。

```ts
function show(a: string) {}

function add(a: number, b: number) {}
```

関数も変数と同様に定義から型が割り当てられる。

```ts
// (a: number, b: number) => number
function add(a: number, b: number) { return a + b }
```

## 制御フロー

引数の型にユニオン型を指定することもできる。  

```ts
function show(x: number | string) {
    return x.split("?")
}
```

このコードはエラーになる。  
`x` は数値の可能性があるため、型検証の結果 `split` プロパティを持たない型を指定しているエラーが発生する。

文字列であることのチェックを盛り込むことで、このエラーは解消できる。

```ts
function show(x: number | string) {
    if (typeof x === 'string') {
        return x.split("?")
    }
}
```

`if` ブランチ内部では `x` が文字列型に絞り込まれる。これを**制御フロー**という。  
制御フローの基本パターンは以下の4つ。

### 1. 等価演算子
リテラル型を絞り込む場合は等価演算子を利用する。

```ts
function show(x: 1 | 2 | 3) {
    if (x === 1) {
        // 1型
    } else {
        // 2 | 3型
    }
}
```

### 2. typeof
基本型を絞り込む場合は `typeof` を利用する。

```ts
function show(x: number | string) {
    if (typeof x === 'string') {
        // string
    } else {
        // number
    }
}
```

### 3. instanceof
`typeof null` では `'object'` が返る。  
`null` を含むユニオン型の場合は `instanceof` を利用する。

```ts
function show(x: Date | null) {
    if (x instanceof Date) {
        // Date
    } else {
        // null
    }
}
```

### 4. in
オブジェクトリテラルの絞り込みにはプロパティの存在チェックとして `in` を利用する。

```ts
function show(x: {id: number} | {name: string}) {
    if ('id' in x) {
        // {id: number}
    } else {
        // {name: string}
    }
}
```

## 構造的部分型
今まで取り上げた通り、引数の型は**事前に定義したものでなくとも**問題ない。  

```ts
type T = {id: number};

function show1(x: T) {}
function show2(x: { id: number }) {}

// 引数の型：定義済の型
// 割り当てる値：無名の型
show1({id: 42}) // ビルドが通る

const a: T = {id: 42}

// 引数の型：無名の型
// 割り当てる値：定義済の型
show2(a) // これでもビルドが通る
```

つまり、構造が同じであれば定義済の型と無名の型は区別されない。  
このシステムを**構造的部分型**という。

一方で、Javaなどは型同士で構造が同じでも継承関係になければ両者は区別される（互換性を持たない）。  
これを**公称型**という。

構造的部分型は、公称型と比較して「型名はデータ構造の別名に過ぎない」（継承関係に基づく互換性の役割を持たない）とも言える。

現状では「使用範囲が狭い型を定義しなくてよい」程度の特徴だが、  
型制約を使う上で覚えておきたい特性。

## 制御フローを理解していない世界
条件分岐を持つ値の型を正確に表現できない場合のデメリットが一層大きくなる。

```ts
function showx(a: 1 | 2 | 3 | 4 | 5) {
    if (a === 1 || a === 2) {
        // 1 | 2 型
    } else {
        // 3 | 4 | 5型
    }
}

function showy(a: number) {
    if (a === 1 || a === 2) {
        // 1 | 2 型
    } else {
        // number型
    }
}
```

```ts
function showz(a: string | number | boolean) {
    if (typeof a === 'string') {
        // string
    } else {
        // number | boolean
    }
}

function showz(a: any) {
    if (typeof a === 'string') {
        // string
    } else {
        // any
    }
}
```

単に引数を自己文書化できないだけでなく、ブランチ内で静的に型を絞り込まれない。  
型の検証を行うのがTypeScriptのメリットであるため、  
型が絞り込まれるようにユニオン型を積極的に活用する。

## ここまでのまとめ
- 条件分岐を行う値の型を正確に表現することで、ブランチごとに絞り込まれた型に推測できる
    + ユニオン型を活用する
    + 基本的な制御フローは4パターン
- 構造的部分型を採用している
    + 事前に型を定義しなくてよい
    + 型名はデータ構造の別名に過ぎない

## ここではできないこと
- ユニオン型の構成要素が複雑な場合に対応できない
- 今まで定義してきた型の再利用
